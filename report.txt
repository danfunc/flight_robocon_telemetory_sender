= 一言でいうと

Shizuku は、私が自作した *「オブジェクト指向マイクロカーネル」* です。
最大の特徴は、*MMU（メモリ保護ユニット）を一切使わずに、ARM の SVC 例外だけで
オブジェクト境界をまたぐ「保護されたメソッド呼び出し」を実現している* ところにあります。

== 最大の特徴：SVC 例外を IPC トラップにした「オブジェクト境界」

一般的なマイクロカーネルは、プロセスごとに仮想アドレス空間を割り当て、MMU で
メモリを物理的に分離します。Shizuku はそれをやりません。全オブジェクトが
*単一の物理アドレス空間* を共有します。代わりに、

- システム全体を *オブジェクト（0–127 のスロット）* と *スレッド（0–127 のスロット）* の
  2 つを基本単位として構成し、
- オブジェクトをまたぐ呼び出しを、すべて *ARM の `SVC` 例外* に集約します。

呼び出しが発生すると、カーネルは例外ハンドラの中で

+ 呼び出し元のフレームを `call_stack` に退避し、
+ プログラムカウンタ（`pc`）を相手オブジェクトの `method_table[m]` へ張り替え、
+ 「現在のオブジェクト ID」を相手オブジェクトへ付け替える

という操作を行います。これにより、呼び出された側のコードは
*同一スレッド上で、相手オブジェクトの権限で* 実行されます。
戻り（`METHOD_EXIT`）では `call_stack` を pop して、元の `pc` とオブジェクト ID を
復元します。つまり *「関数呼び出しと全く同じ感覚で、カーネルが境界と権限を保証した
サブルーチン呼び出し」* を書けるのが核心です。

MMU を使わないので *ポインタをそのまま共有できる* のも強みです。大きなデータは
アドレスだけを引数で渡し、受け手がコピーする — というゼロコピーに近い受け渡しが
可能になります（戻り値はレジスタ `r1` に 1 つ載せて返す）。

== 2 層の SVC と権限委譲

特権の与え方も特徴的です。SVC を 2 層に分けています。

/ カーネルオブジェクト (id=0): `METHOD_CALL` / `METHOD_EXIT` /
  `SWITCH_THREAD` などの低位プリミティブを直接実行できる特権オブジェクト。
/ 一般オブジェクト: `CREATE_OBJECT` や `CALL_METHOD`、`YIELD` といった高位 API を
  すべて `svc 0x00` で発行する。

一般オブジェクトが `svc 0x00` を撃つと、カーネルはそれを直接処理せず、
*登録済みの SVC ハンドラ（＝カーネルオブジェクト）へトランポリン* します。
このとき元の SVC 番号・呼び出し元のオブジェクト/スレッドを callee-saved な
`r4`/`r5`/`r6` に載せて渡し、*同じスレッドのままカーネルオブジェクトの権限へ昇格*
させます。「権限はカーネルが、その都度、呼び出しの形で委譲する」という設計で、
ユーザ拡張（SVC ハンドラ）をカーネル空間に安全に差し込めます。

== スケジューリング

プリエンプションは持たない *協調型 (cooperative)* です。各スレッドが明示的に
`yield`（READY な次スレッドを探して `SWITCH_THREAD`）します。
I2C アクセスや SVC 処理は、自分で yield しない限り原子的に走るため、
ロックなしでも一貫性が保ちやすいのが利点です。

== 設計上の落とし穴：FP 例外フレーム非対応

Shizuku の例外フレームは *FPU レジスタを退避しません*。そのため、ハードフロート
の libm（`sqrtf`/`powf` など）を踏むと FPU の FPCA ビットが立ち、コンテキスト
スイッチが壊れて HardFault に至ります。これを避けるため、浮動小数点を使うオブジェクト
では FPU を踏まない自作の `include/soft_math.hpp`（`sqrtf`/`fabsf`）を使っています。
— これは「単一アドレス空間・軽量例外フレーム」という設計判断の裏返しでもあります。

= デモの説明：BLE テレメトリ送信システム

このデモは、Shizuku のオブジェクトモデルが *実機の飛行テレメトリ* で動くことを示す
ものです。各ハードウェアドライバを独立した Shizuku オブジェクトとして実装し、
メソッド呼び出しだけで連携させています。

== オブジェクト構成

ブートすると、カーネルオブジェクトが `IO_CONTROLLER` を起動し、そこから順に
各オブジェクトが生成されます。

#table(
  columns: (auto, 1fr),
  inset: 6pt,
  align: (left, left),
  stroke: 0.5pt + gray,
  table.header([*オブジェクト*], [*役割*]),
  [`IO_CONTROLLER`], [起動の起点。各ドライバ／センサ／送信オブジェクトを生成する。],
  [`BLE_UART_DRIVER`], [BLE の Nordic UART Service (NUS) を張り、`send_byte` で送信、受信は `rx_sink` 経由でコールバックする。],
  [`BME280_DRIVER`], [I2C 気圧・温度センサ。50Hz で読み続け、最新値をキャッシュ。`read_latest` で購読できる。],
  [`BNO055_DRIVER`], [I2C 9 軸 IMU。姿勢（roll/pitch/heading）・線形加速度・較正状態を `read_latest` で提供。],
  [`TELEMETRY_SENDER`], [上記 2 センサを購読し、相補フィルタで高度・上下速度を融合、BLE 経由で母艦へ送る中核オブジェクト。],
  [`WS2812_DRIVER`], [ステータス表示用の LED ドライバ。],
)

センサとアプリケーションが *別オブジェクトに分かれていて、`call_method` でしか
互いに触れない* 点が重要です。`read_latest` には受け手のバッファのアドレスを
渡し、スナップショットをコピーして受け取ります — まさに「単一アドレス空間 ＋
ポインタ渡し」の例です。

== 信号処理（TELEMETRY_SENDER の中身）

50Hz でセンサを回し、次の処理をします。

- *慣性予測*：傾き補正済みの「世界鉛直加速度」を積分して高度・上下速度を予測。
- *相補フィルタ*：BME280 の気圧高度が更新されるたびに融合（係数 `ALPHA_H=0.80`,
  `ALPHA_V=0.75`）して、ドリフトとノイズを両取り。
- *状態判定*：ヒステリシス付きで「水平／上昇／下降」を判定。
- *高度保持 PID*：目標高度 1.0m に対するサーボ指令を算出（監視用）。

== 送信路の工夫：バイナリ・バッチフレーム

BLE は「1 接続イベントあたり実質 1 notify」が上限で、サンプルを 1 個ずつ送ると
事象数（約 33/s）に律速されます。そこで、

+ センサ融合を高レートで回してサンプルを内部リングに溜め、
+ 複数サンプル（最大 12 件）を 1 フレームにまとめて送る *バッチ送信* にし、
+ ワイヤ表現は `batch_header_t(17B) + サンプル(35B)×N + CRC16-CCITT(2B)` の
  *バイナリ*（CSV の約 1/3）として、
+ 改行 `0x0A` をバイトスタッフィングし、先頭マーカー `0x02` で括ることで、
  BLE_UART の「`'\n'` で 1 行を原子的にコミット」という性質に乗せて、
  *フレームが丸ごと届くか丸ごと落ちるか（部分受信なし）* を保証しています。

これにより、イベント数の壁を越えて *ロスのない高レートテレメトリ* を実現しています。

母艦からは BLE 経由で 1 文字コマンドを送れます（受信は `rx_sink` で
`TELEMETRY_SENDER` に届く）。

= まとめ（自己紹介のキーメッセージ）

- *Shizuku の最大の特徴*：MMU なしの単一アドレス空間で、ARM の SVC 例外を IPC
  トラップに使い、`pc` とオブジェクト ID を張り替えることで「カーネルが保証した
  オブジェクト境界をまたぐメソッド呼び出し」を実現したマイクロカーネル。
- *デモ*：そのカーネル上で、センサ／通信ドライバを独立オブジェクトとして実装し、
  相補フィルタによるセンサ融合と、BLE のイベント律速を越えるバイナリ・バッチ送信で、
  実機の飛行テレメトリをロスなく届けるシステムを構築した。
